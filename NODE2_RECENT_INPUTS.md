# Node2 Recent Inputs

以下内容用于复测 Node2 的表现，来源于：

- 生产环境当前代码中的 Node2 system prompt / user prompt 模板
- `chat_messages` 中的真实消息
- `chat-worker` 日志中的 batch / recall 信息
- Qdrant 中可确认的旧 impression 内容

说明：

- 下面给的是最近两次可还原的 Node2 输入。
- `batch=1_2026-04-10_1775807585096` 基本可以完整还原。
- `batch=1_2026-04-10_1775807831746` 中，`id=79b27cb9-5222-4bad-be0a-25636904ccb9` 这条旧 impression 在该 batch 之后被更新了；它在进入该次 Node2 前的 `scene` 和 `points` 可确认，但 `retrievalText` 没有单独落日志，下面按当时已知 `scene + points` 做了重建，并明确标注。

## System Prompt

```text
你是聊天印象系统的 Node2 聊天印象合并器。你根据当前聊天原文和筛选后的旧 impressions，输出最终落库的 scene、points、entities、retrievalText。输出必须是纯 JSON。
```

这是 2026-04-10 最早那批“搬砖”对话，Worker 日志显示：

- `Messages: 2`
- `New messages: 2`
- `Recall: 1`

对应的 user prompt 为：

```text
## 当前聊天原文
[用户] 什么是搬砖
[AI] 哈哈就是上班干活呀～打工人自嘲的说法，把工作比作搬砖啦😜

## 候选旧 impressions
- id=ab301bd3-01ed-478f-83f6-480d2a2c63ca; scene=聊用户早晨到公司后的早餐内容、习惯及公司免费热水的情况; points=["用户早餐习惯固定，每天早上吃两个鸡蛋和一个肉包子","用户提到豆浆价格3元且认为营养价值一般","用户表示公司可能没有免费热水（用户反馈‘我感觉没有’）"]; retrievalText=用户早晨到公司后吃早餐，习惯每天两个鸡蛋和一个肉包子，提到豆浆价格3元且认为营养价值一般；用户还表示公司可能没有免费热水。; memoryDate=2026-04-08; originType=standalone; sourceImpressionId=null; rootImpressionId=ab301bd3-01ed-478f-83f6-480d2a2c63ca

你是 Node2：聊天印象合并器。你只输出最终要落库的聊天印象。
请按下面 3 个阶段完成，但不要输出中间过程，只输出最终 JSON。

阶段 1：先判场景
- 先只根据当前聊天原文判断这轮主要在聊什么，不要先被旧 impressions 带偏。
- 默认只输出 1 条 impression。
- 只有明确、独立、目标完全不同的多场景切换，才允许输出多条。
- 如果多个内容只是围绕同一个主问题的补充、举例、纠正或分支展开，仍然视为同一条 impression。
- 如果合并后已经覆盖多个独立目标、对象或矛盾，就拆开，避免生成巨 impression。

阶段 2：再判关系
- 对每个场景判断是否承接某条旧 impression；只有明显属于同一主线时才填写 sourceImpressionId，否则填 null。
- 如果旧 impression 和当前原文不一致，以当前原文为准。
- 如果是承接旧 impression，先判断当前信息主要是补充还是冲突：
  - 补充时，优先保留原有事实，只补充新的客观信息。
  - 冲突时，不要抹掉旧事实；在新的 point 中明确写出“现在和之前不一致/发生变化”。
- 不要把暂时情绪、一次性表态或证据不足的内容写成稳定变化。

阶段 3：最后生成
- 先生成 points：
  - 只保留真正持久的记忆点，优先保留关键互动、关键纠正、关键争议、关键变化和客观事实，少写空泛总结。
  - 除非这一轮几乎只有一方在持续输出，否则 points 必须尽量体现双方内容，不能只留下单边表述。
  - 只写有明确聊天证据支持的内容，不要根据常识、身份、语气去推测用户没说出的事实。
  - 去分析化，不按时间线复盘；尽量使用第一视角，但保持客观克制。
- 再生成 scene：
  - 只写“我和用户在聊什么”，不要把结论、纠正、分析、建议写进 scene。
- 再抽取 entities：
  - 只保留最关键的客观锚点，优先保留人物、关系身份、具体对象、地点、时间。
- 最后生成 retrievalText：
  - 只能基于最终 scene + points + entities 生成，不得新增它们里没有的新事实。
  - 多保留实体和客观事实，少保留分析性空话。
- 输出前自检：
  - 不要误把两个独立话题硬合成一条 impression。
  - 不要错误覆盖旧事实。
  - 不要偏离当前原文。
  - 不要遗漏明显重要的双方互动。
  - 不要写入没有明确证据支持的推测。

输出要求：
- 直接输出 JSON。

输出格式：
{"impressions":[{"sourceImpressionId":"old_id_or_null","scene":"...","points":["..."],"entities":["..."],"retrievalText":"..."}]}
```
